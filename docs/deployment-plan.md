# План Вывода В Прод

## 1. Как Устроены Расходы

- `VPS` (`удаленный сервер`) обычно оплачивается `ежемесячно`.
- `Домен` (`красивый адрес сайта`) обычно оплачивается `раз в год`.
- `Субдомены` (`api.example.com`, `client1.example.com`) обычно бесплатны внутри одного домена.

Простой вывод:
- постоянная ежемесячная трата у вас в основном одна: `VPS`
- домен обычно покупается один раз на год и потом продлевается

## 2. Минимальный Бюджет Для Старта

Что нужно:
- `1 VPS`
- `1 домен`

Ориентир:
- `VPS`: примерно `$4-7` в месяц
- `Домен`: примерно `$10-18` в год

Это значит:
- не нужно покупать отдельный домен каждому клиенту
- не нужно поднимать отдельный сервер под каждого клиента на старте

## 3. Самая Дешевая Рабочая Схема

Текущая логика проекта:
- бот + API + scheduler запускаются одной командой `python main.py`
- WebApp уже отдельно живет на GitHub Pages

Самая практичная схема:
- WebApp: оставить на GitHub Pages
- API: перенести на VPS
- бот: перенести на VPS
- база `bookings.db`: хранить на VPS
- `ngrok`: убрать полностью

Как это будет выглядеть:
- WebApp: `https://username.github.io/project`
- API: `https://api.yourdomain.com/api`

## 4. Что Означают Основные Части

### VPS

`VPS` это ваш удаленный компьютер в интернете, который работает 24/7.

Зачем нужен:
- бот не зависит от вашего ПК
- не нужен домашний интернет
- не нужен `ngrok`

### systemd

`systemd` это менеджер служб в Linux.

Проще:
- он запускает вашего бота как постоянный сервис
- если бот упал, он поднимет его снова
- если сервер перезагрузился, он запустит его автоматически

### nginx

`nginx` это веб-сервер и `reverse proxy` (`прокладка` между интернетом и вашим приложением).

Проще:
- запрос приходит на домен
- `nginx` принимает его
- передает дальше в FastAPI

### SSL

`SSL` / `TLS` это защищенное соединение, то есть `https`.

Проще:
- без этого продовый WebApp и API выглядят неправильно
- через `https` данные идут безопасно

## 5. Нужен Ли Отдельный Домен Каждому Клиенту

Нет.

Это важный момент:
- покупается `один домен`
- внутри него можно делать много субдоменов

Пример:
- `api.myservice.com`
- `client1.myservice.com`
- `client2.myservice.com`

Обычно отдельно за каждый такой адрес платить не нужно.

## 6. Как Дешево Запуститься

### Этап 1. Один Свой Бот

Что делаете:
- покупаете `1 VPS`
- покупаете `1 домен`
- переносите текущий проект на сервер
- API вешаете на `api.вашдомен.com`

Это самый правильный старт.

### Этап 2. Первые Клиенты

Если появятся первые 1-3 клиента, не делайте сразу SaaS-платформу.

Можно проще:
- один и тот же VPS
- для каждого клиента отдельная папка проекта
- у каждого клиента свои:
  - `.env`
  - `config.json`
  - `bookings.db`
  - `BOT_TOKEN`

Пример структуры:

```text
/opt/client-a-bot
/opt/client-b-bot
/opt/client-c-bot
```

То есть:
- один сервер
- один домен
- несколько отдельных экземпляров

Это дешево и нормально для раннего этапа.

## 7. Когда Придется Переделывать Архитектуру

Сейчас проект фактически `single-tenant` (`один проект = один клиент`), потому что у вас:
- один `BOT_TOKEN`
- один `config.json`
- одна база `bookings.db`
- один `ADMIN_ID`

Переделка нужна будет, когда:
- клиентов станет больше 3-5
- станет неудобно руками держать много папок
- захотите массово раздавать trial
- захотите централизованно управлять клиентами

До этого можно жить на отдельных экземплярах.

## 8. Что Я Рекомендую Именно Сейчас

Не делать пока сложную multi-client систему.

Лучший путь:
1. Поднять один экземпляр на VPS
2. Убрать `ngrok`
3. Проверить, что все стабильно работает на сервере
4. Дать продукт первым людям
5. Посмотреть, нужен ли он вообще рынку
6. Только потом думать о платформе для многих клиентов

## 9. Что Купить Сначала

В правильном порядке:
1. `VPS`
2. `Домен`
3. Настроить сервер
4. Прописать DNS
5. Поднять API и бота
6. Поменять фронт с `ngrok` на ваш постоянный API URL
7. Протестировать все заново

## 10. На Что Смотреть По Ценам

Примеры VPS-провайдеров:
- `Hetzner`
- `Contabo`
- `DigitalOcean`

Примеры регистраторов доменов:
- `Namecheap`
- `Porkbun`
- `Cloudflare Registrar`

Важно:
- домен обычно оплачивается `не помесячно`, а `на год`
- VPS обычно оплачивается `помесячно`

## 11. Короткий Вывод

Для вашего бюджета реалистичный старт такой:
- `1 VPS`
- `1 домен`
- без отдельных доменов на каждого клиента
- без отдельного сервера на каждого клиента
- сначала один боевой экземпляр
- потом несколько отдельных экземпляров на том же VPS

Это самый дешевый и здравый путь для проверки спроса.
